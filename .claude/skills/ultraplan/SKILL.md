# Ultra Plan — 심층 구현 계획 수립

사용자가 `/ultraplan <작업 설명>`을 입력하면, 해당 작업에 대해 **심층 분석 → 계획 수립 → 리스크 평가 → 실행 계획**을 자동으로 생성합니다.

## 사용법
```
/ultraplan migrate the auth service from sessions to JWTs
/ultraplan 게시판 시스템에 실시간 알림 기능 추가
/ultraplan Pixel Streaming 로컬 테스트 환경 구성
```

## 실행 프로세스

### Phase 1: 컨텍스트 수집 (Explore)

**병렬로 최대 3개의 Explore 에이전트**를 실행하여 코드베이스를 분석합니다.

#### 에이전트 1 — 영향 범위 분석
- 작업 대상과 관련된 **파일, 모듈, 의존성** 탐색
- 기존 구현 패턴, 유틸리티, 재사용 가능한 코드 식별
- 관련 설정 파일 (config, env, package.json, requirements.txt 등) 확인

#### 에이전트 2 — 아키텍처 분석
- 현재 시스템 아키텍처에서 작업이 영향을 미치는 레이어 파악
- 데이터 흐름 추적 (프론트엔드 → API → DB)
- 기존 패턴과의 일관성 확인

#### 에이전트 3 — 리스크 & 의존성 분석
- 작업으로 인해 깨질 수 있는 기존 기능 식별
- 외부 의존성 (라이브러리, API, 서비스) 확인
- 마이그레이션 필요 여부 (DB 스키마 변경, 데이터 이전 등)

> 단순한 작업의 경우 에이전트 1개로 충분합니다. 작업 복잡도에 따라 판단하세요.

### Phase 2: 계획 수립 (Plan)

Phase 1 결과를 기반으로 **Plan 에이전트**를 실행하여 구현 계획을 수립합니다.

계획에 포함해야 할 항목:

#### 2-1. 개요
```markdown
## 개요
- **작업**: (사용자 요청 요약)
- **복잡도**: Simple / Medium / Complex
- **예상 파일 수**: N개
- **리스크 등급**: Low / Medium / High
```

#### 2-2. 사전 조건
```markdown
## 사전 조건
- [ ] (필요한 패키지 설치)
- [ ] (환경변수 설정)
- [ ] (마이그레이션 필요 여부)
```

#### 2-3. 단계별 실행 계획
```markdown
## 실행 계획

### Step 1: (제목)
- **파일**: `path/to/file.py`
- **작업**: (구체적 변경 내용)
- **검증**: (이 단계 완료 확인 방법)

### Step 2: (제목)
...
```

각 Step에는 반드시 포함:
- **대상 파일** 경로 (절대 경로)
- **구체적 변경 내용** (함수명, 클래스명 등 구체적으로)
- **재사용할 기존 코드** (Phase 1에서 발견한 유틸, 패턴)
- **검증 방법** (테스트, curl, 브라우저 확인 등)

#### 2-4. 리스크 평가
```markdown
## 리스크 평가

| 리스크 | 등급 | 완화 방안 |
|--------|------|-----------|
| (리스크 설명) | P0~P3 | (대응 방법) |
```

- **P0 (Critical)**: 서비스 중단, 데이터 손실 가능 → 반드시 해결 후 진행
- **P1 (High)**: 주요 기능 영향 → 우선 대응
- **P2 (Medium)**: 부분적 영향 → 일정 내 대응
- **P3 (Low)**: 미미한 영향 → 모니터링

#### 2-5. 롤백 계획
```markdown
## 롤백 계획
- git revert 가능 여부
- DB 마이그레이션 롤백 방법
- 기능 플래그 비활성화 방법
```

#### 2-6. 검증 체크리스트
```markdown
## 검증 체크리스트
- [ ] 로컬 테스트 통과
- [ ] 기존 기능 영향 없음 확인
- [ ] 배포 환경 호환성 확인
- [ ] (추가 검증 항목)
```

### Phase 3: 사용자 검토

계획을 사용자에게 **마크다운 형식**으로 보여주고 피드백을 받습니다.

```
📋 Ultra Plan이 완성되었습니다.

[계획 내용 표시]

이대로 실행할까요? 수정이 필요한 부분이 있으면 알려주세요.
```

사용자 피드백 선택지:
1. **실행** → Phase 4로 진행
2. **수정** → 피드백 반영 후 계획 업데이트
3. **취소** → 종료

### Phase 4: 실행 (Execute)

사용자 승인 후, 계획의 각 Step을 **순서대로** 실행합니다.

실행 규칙:
1. 각 Step 시작 전 TodoWrite로 진행 상태 표시
2. 각 Step 완료 후 **검증** 수행
3. 검증 실패 시 → 즉시 중단, 원인 분석, 사용자에게 보고
4. 모든 Step 완료 후 → 최종 검증 체크리스트 수행
5. 커밋은 사용자 지시가 있을 때만 수행

## 계획 모드 선택 기준

| 모드 | 조건 | Phase 1 에이전트 수 |
|------|------|---------------------|
| **Simple** | 파일 1~3개 수정, 리스크 없음 | 1개 |
| **Standard** | 파일 4~10개, 중간 복잡도 | 2개 |
| **Deep** | 10개+ 파일, 아키텍처 변경, DB 마이그레이션 | 3개 |

작업 설명을 분석하여 자동으로 모드를 선택합니다. 사용자가 명시할 수도 있습니다:
```
/ultraplan --deep implement real-time notifications with WebSocket
/ultraplan --simple fix the login button color
```
