# Code Review

변경된 코드에 대해 체계적인 코드 리뷰를 수행하고 등급별 리포트를 생성합니다.
코드를 수정하지 않고 **문제점을 문서화**하는 것이 목적입니다.

## 1단계: 리뷰 범위 결정

인자가 주어지면 해당 범위를 사용합니다:

| 인자 | 동작 |
|------|------|
| `staged` | `git diff --cached` 스테이징된 변경사항 |
| `last-commit` | `git diff HEAD~1` 마지막 커밋 |
| `<commit>..<commit>` | 지정된 커밋 범위 |
| `<파일경로>` | 해당 파일만 리뷰 |
| (없음) | `git diff` 미커밋 변경사항 + `git diff --cached` 스테이징 변경사항 |

```bash
# 범위에 맞는 diff를 수집
git diff [범위] --stat  # 변경 파일 목록
git diff [범위]          # 전체 diff
```

변경된 파일이 없으면 "리뷰할 변경사항이 없습니다"를 출력하고 종료합니다.

## 2단계: 변경 파일 분석

변경된 각 파일의 전체 내용을 Read 도구로 읽어 문맥을 파악합니다.
diff만 보지 말고 **주변 코드와의 관계**도 반드시 확인합니다.

## 3단계: 5개 차원 점검

### 3-1. 보안 (Security)

- **SQL Injection**: raw query, 미검증 사용자 입력
- **XSS**: 미이스케이프 출력, dangerouslySetInnerHTML
- **인증/인가**: 누락된 권한 체크, 하드코딩된 시크릿
- **의존성**: 알려진 취약 패키지
- **CORS/CSRF**: 잘못된 설정
- **파일 업로드**: 미검증 파일 타입/크기

### 3-2. 성능 (Performance)

- **N+1 쿼리**: 루프 내 DB 호출
- **불필요한 렌더링**: 누락된 memo/useMemo/useCallback
- **번들 사이즈**: 불필요한 import, tree-shaking 방해
- **비동기 처리**: 블로킹 호출, 누락된 에러 핸들링
- **캐싱**: 반복 연산, 미활용 캐시

### 3-3. 코드 품질 (Quality)

- **DRY 위반**: 복붙 코드, 추출 가능한 공통 로직
- **복잡도**: 깊은 중첩, 긴 함수 (30줄+), 복잡한 조건
- **네이밍**: 모호한 변수명, 일관성 없는 네이밍
- **에러 처리**: 빈 catch, 무시된 에러, 부적절한 fallback
- **타입 안전성**: any 남용, 누락된 타입 가드

### 3-4. 베스트 프랙티스 (Best Practices)

- **프레임워크 관례**: React hooks 규칙, FastAPI 패턴 준수
- **프로젝트 컨벤션**: CLAUDE.md 규칙 준수 여부
- **API 설계**: RESTful 원칙, 일관된 응답 형식
- **DB 패턴**: ORM 사용법, 마이그레이션 안전성
- **테스트**: 테스트 커버리지, 엣지 케이스

### 3-5. 유지보수성 (Maintainability)

- **모듈화**: 적절한 파일/함수 분리
- **의존성 방향**: 순환 참조, 잘못된 import 방향
- **문서화**: 복잡한 로직의 주석 누락
- **확장성**: 하드코딩된 값, 매직 넘버

## 4단계: 심각도 분류

각 이슈를 아래 등급으로 분류합니다:

| 등급 | 의미 | 조치 |
|------|------|------|
| 🔴 P0 | **치명적** — 보안 취약점, 데이터 손실 가능 | 즉시 수정 필수 |
| 🟠 P1 | **높음** — 버그, 성능 심각 저하 | 머지 전 수정 권장 |
| 🟡 P2 | **보통** — 코드 품질, 컨벤션 위반 | 후속 작업으로 수정 |
| 🟢 P3 | **낮음** — 개선 제안, 스타일 | 선택적 개선 |

## 5단계: 리포트 출력

아래 형식으로 리포트를 생성합니다:

```markdown
## 코드 리뷰 리포트

### 리뷰 범위
- **변경 파일**: N개
- **추가/삭제**: +N / -N 줄
- **범위**: (커밋 범위 또는 파일 목록)

### 종합 점수

| 차원 | 점수 | 상태 |
|------|------|------|
| 보안 | ★★★★☆ | 양호 |
| 성능 | ★★★☆☆ | 주의 |
| 코드 품질 | ★★★★☆ | 양호 |
| 베스트 프랙티스 | ★★★★★ | 우수 |
| 유지보수성 | ★★★★☆ | 양호 |

**종합**: ★★★★☆ (4.0/5.0)

### 이슈 목록

#### 🔴 P0 — 치명적
(없으면 "발견된 P0 이슈 없음 ✅" 출력)

#### 🟠 P1 — 높음
| # | 파일 | 줄 | 이슈 | 설명 |
|---|------|-----|------|------|
| 1 | file.py:42 | 42 | 제목 | 상세 설명 |

#### 🟡 P2 — 보통
(같은 테이블 형식)

#### 🟢 P3 — 낮음
(같은 테이블 형식)

### 잘한 점 👍
- (긍정적인 부분도 반드시 언급)

### 권장 조치
1. (P0/P1 이슈 수정 방법)
2. (P2 후속 작업 제안)
```

## 규칙

1. **코드를 수정하지 않는다** — 리포트만 생성
2. **구체적으로 지적한다** — 파일명:줄번호 포함, "뭔가 이상함" 금지
3. **근거를 제시한다** — 왜 문제인지, 어떤 영향이 있는지 설명
4. **긍정적 피드백도 포함** — 잘한 점을 반드시 1개 이상 언급
5. **프로젝트 맥락을 고려** — CLAUDE.md, 기존 코드 패턴 참조
6. **과도한 지적 금지** — 스타일 취향 차이는 P3로, 실제 문제에 집중
